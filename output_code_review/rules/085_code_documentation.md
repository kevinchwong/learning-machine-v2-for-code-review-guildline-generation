# Rule 1. Ensure docstrings are written in English

| Frequent score for this rule: 100.0 | [^Top](#code-documentation) 

## Explanation
>Code Documentation Rule: Ensure docstrings are written in English to maintain consistency and readability.

## Why do we need this rule?
>Using English docstrings ensures that all team members can easily understand and maintain the codebase, regardless of their native language. It also facilitates collaboration and reduces the risk of misinterpretation or confusion.

## If not apply this rule, what will happen?
| The docstring is written in French, which can create confusion for team members who do not understand the language, leading to misinterpretation and difficulties in maintaining the codebase.
```python
def multiply(a, b):
    """
    Multiplie deux nombres et retourne le résultat.
    
    Args:
        a (int): Le premier nombre.
        b (int): Le deuxième nombre.
    
    Returns:
        int: Le produit des deux nombres.
    """
    return a * b
```

## If apply this rule?
| The docstring is written in English, providing clear information about the function's purpose, arguments, and return value, enhancing code readability and maintainability.
```python
def add_numbers(a, b):
    """
    Add two numbers and return the result.
    
    Args:
        a (int): The first number.
        b (int): The second number.
    
    Returns:
        int: The sum of the two numbers.
    """

```

# Rule 9. Document the expected behavior of functions under edge cases

| Frequent score for this rule: 85.0 | [^Top](#code-documentation) 

## Explanation
>Code Documentation should include the expected behavior of functions under edge cases to provide clarity on how the function should behave in unusual or extreme scenarios.

## Why do we need this rule?
>This rule ensures that developers have a clear understanding of how functions should handle edge cases, reducing the risk of unexpected behavior and improving code reliability and maintainability.

## If not apply this rule, what will happen?
| This code does not document the behavior when dividing by zero, leading to ambiguity and potential errors for developers.
```python
def divide_numbers(num1, num2):
    if num2 == 0:
        return None
    return num1 / num2
```

## If apply this rule?
| The function 'divide_numbers' includes documentation specifying the behavior when dividing by zero, providing clarity for developers.
```python
def divide_numbers(num1: int, num2: int) -> float:
    """
    Divide two numbers and return the result.
    Edge cases:
    - If num2 is 0, raise a ZeroDivisionError.
    """
    if num2 == 0:
        raise ZeroDivisionError("Cannot divide by zero")
    return num1 / num2
```
